# Playback Queue Design

This document records the intended Easy Music playback queue model. It is a
product and architecture spec for future queue work, not a statement that all
behavior below is already implemented.

## Purpose

Easy Music now has ordinary persisted playlists. The next step is to separate
that durable organization model from the temporary "what is playing now" model.

The product should have two distinct concepts:

- **Playlist**: a persisted, owner-scoped list of tracks used for long-term
  library organization.
- **Playback Queue**: a local, temporary playback session used to decide what
  plays now, next, and later in the current client.

Playlists can generate queues. Queues never modify playlists.

## Non-Goals

- No server-side playback queue model.
- No `/api/queues` endpoints.
- No cross-device queue sync.
- No long-term queue persistence.
- No queue-to-playlist save action.
- No smart playlists.
- No multi-playlist combined queue looping.
- No single-track repeat.
- No generic list repeat for non-playlist queues.
- No recommendation algorithm changes.

## Persistence Boundary

The queue is local and temporary:

- Web keeps queue state in the current page/app session.
- Android keeps queue state in the current player service/process lifetime.
- Browser refresh, tab close, app process death, or clearing the app from the
  background may discard the queue.
- The backend remains responsible only for durable playlists, track metadata,
  ordered playlist detail, and authenticated streaming.

## Queue State Model

A queue is made from queue items, not directly from track ids. This allows the
same track to appear multiple times in one queue.

Recommended logical shape:

```text
PlaybackQueue
  id
  source
  generationMode
  repeatPlaylist
  history[]
  current
  upcoming[]
  baseCycleItems[]
```

### Queue Item

Each queue item should include:

- `queueItemId`: local unique id for this queue item.
- `track`: the track payload required by the player.
- `origin`: optional source details, such as playlist id or manual insertion.
- `cycleItem`: whether this item belongs to the playlist cycle base.

Operations such as remove and drag reorder target `queueItemId`, not `trackId`.

### History

`history` is the actual playback history for the current queue. It is used by
the previous-track command.

History is not shown as an editable first-version UI list.

### Current

`current` is the track currently loaded or playing.

The current item is fixed and is not part of drag reorder.

### Upcoming

`upcoming` is the editable list of items scheduled after the current item.

The first version supports:

- Insert next.
- Add to queue tail.
- Remove one queue item.
- Clear the whole queue.
- Drag reorder upcoming items.

Drag reorder applies only to `upcoming`, never to `history` or `current`.

### Base Cycle Items

`baseCycleItems` exists only when a queue is generated from one source playlist.
It represents the playable playlist tracks used to generate future loop rounds.

Manually inserted tracks are not added to `baseCycleItems`.

## Queue Sources

Supported source types:

- `playlist`: generated from one playlist.
- `singleTrack`: generated by immediate play from one track.
- `manual`: built by user queue edits.
- `recommendation`: reserved for future recommendation playback, not part of
  the first implementation.

Only `playlist` queues can use playlist repeat.

## Generation Modes

Generation modes are applied when creating a queue. They are not dynamic modes
that reshuffle or reverse a running queue.

Supported playlist generation modes:

- `sequence`: use playlist order.
- `shuffleOnce`: shuffle once at queue creation. A round does not repeat tracks.
- `reverse`: use reverse playlist order.

If the user wants a different mode, they start a new queue from the playlist.

## Track Eligibility

Queue generation filters out non-playable tracks:

- `ready` tracks enter the queue.
- `uploaded`, `processing`, `failed`, and other non-ready statuses are skipped.
- The UI should report how many tracks were skipped.
- If no ready tracks remain, no queue is created and the UI shows a clear
  "playlist has no playable tracks" message.

If a ready track fails at playback time, the player should show a useful error
and try the next item. If no next item exists, normal end-of-queue behavior
applies.

## Playback Actions

### Immediate Play

Immediate play replaces the current queue with a new single-track queue and
starts playback.

### Play Next

Play next inserts a queue item at the start of `upcoming`. It does not interrupt
the current track.

### Add To Queue

Add to queue inserts a queue item at the end of `upcoming`. It does not
interrupt the current track.

### Playlist Play

Playlist sequence, shuffled, or reverse play replaces the current queue.

If a queue is currently playing, the client should ask for confirmation before
replacing it.

### Next

Next moves the current item to history, then plays the first upcoming item.

If upcoming is empty and playlist repeat is enabled, the client generates the
next playlist round from `baseCycleItems`.

If upcoming is empty and repeat is disabled, playback stops at the end of the
queue.

### Previous

Previous directly plays the last item in history.

There is no "seek to start after 3 seconds" behavior. If history is empty, the
previous button should be disabled.

When moving to the previous item, the old current item returns to the start of
upcoming so playback remains reversible.

### Remove Upcoming Item

Removing an upcoming item affects only the current queue and current playback
round. It does not modify the source playlist.

### Remove Current Item

Removing the current item immediately stops that item and plays the next item.

Priority:

1. Play the first upcoming item.
2. If upcoming is empty and playlist repeat is enabled, generate and play the
   next playlist round.
3. If no next item exists, stop playback and clear current.

The removed current item is not added to history.

### Clear Queue

Clear queue stops playback and clears:

- history
- current
- upcoming
- baseCycleItems
- repeat state

If playback is active, the client should ask for confirmation first.

## Playlist Repeat

Playlist repeat is controlled in the queue management UI.

Rules:

- Repeat is available only for queues generated from one playlist.
- Repeat is unavailable for single-track, manual, mixed, or recommendation
  queues.
- Sequence repeat regenerates each round in the same playlist order.
- Reverse repeat regenerates each round in the same reverse order.
- Shuffle repeat reshuffles each round. A round does not repeat tracks.
- Shuffle repeat should avoid making the next round's first item equal to the
  previous round's last item when practical.
- Temporary manually inserted items play once and do not repeat.
- Removing a base playlist item from the queue affects only the current round.
- Permanent removal should be done by editing the source playlist.

## Source Playlist Changes

The current queue follows source playlist changes only within the same client.
Cross-device updates are not synchronized into an already-running queue.

Same-client rules:

- If a ready track is added to the source playlist, it enters `baseCycleItems`
  for the next repeat round, not the current round.
- If a track is removed from the source playlist, it is removed from
  `baseCycleItems`.
- If the removed track is still in upcoming, remove it from upcoming.
- If the removed track is already in history, leave history unchanged.
- If the removed track is current, allow it to continue playing, but do not
  include it in future rounds.
- If `baseCycleItems` becomes empty, disable repeat after the current item and
  show a clear message.

Non-ready tracks added to the playlist do not need live status watching in the
first implementation. They enter future queues after refresh or a new queue
generation once they are ready.

## Queue Management UI

### Web

Web should expose a global playback queue drawer. It should not be limited to
the playlist detail page.

The drawer should show:

- Queue source and source playlist name when present.
- Generation mode.
- Repeat status and repeat toggle when available.
- Played count summary.
- Upcoming count summary.
- Current item.
- Upcoming items.
- Clear queue action.

Upcoming supports:

- Drag reorder.
- Remove item.
- Add next / add tail from library or track surfaces.

History is not shown as a list in the first version.

### Android

Android should expose queue management from Now Playing, either as a queue tab
or a bottom sheet.

Android behavior must match Web semantics even if implementation details differ
because Media3 owns the native playback queue.

The UI should show current and upcoming, not editable history.

## Cross-Client Consistency

Web and Android must share the same product rules:

- Immediate play replaces the queue.
- Play next inserts at upcoming head.
- Add to queue inserts at upcoming tail.
- Previous uses actual history.
- Drag reorder affects only upcoming.
- Remove and clear do not modify playlists.
- Playlist repeat is only for single-playlist source queues.
- Shuffle repeat reshuffles per round.

UI shape may differ by platform.

## Implementation Notes

The current codebase already has:

- Backend persisted playlists with ordered playlist tracks.
- Web playlist detail playback entry points.
- Android Media3 playback and playlist detail entry points.

Future queue work should build on those pieces but introduce a first-class
client queue module rather than keeping queue state local to playlist detail
pages.

Backend changes are not expected unless playlist detail stops providing stable
ordered tracks.

## Acceptance Themes

A complete implementation should prove:

- Queue core operations work on duplicate track entries by queue item id.
- Single-track immediate play replaces the active queue.
- Play next and add to queue preserve current playback.
- Previous follows actual history.
- Drag reorder changes only upcoming.
- Clear queue stops playback.
- Playlist sequence, shuffle-once, and reverse generation work.
- Playlist repeat is available only for single-playlist queues.
- Shuffle repeat creates new no-repeat rounds.
- Queue edits do not modify source playlists.
- Web and Android match the same behavior rules.
